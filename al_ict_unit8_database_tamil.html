8.1தரவு, தகவல், கட்டமைப்பு & தரவுத்தள மாதிரிகள்

தரவு (Data) vs தகவல் (Information)

இது அடிப்படை — தரவு செயலாக்கப்படாத மூலக் கூறு; தகவல் தரவைச் செயலாக்கி அர்த்தமுள்ளதாக்கிய விளைவு. எ.கா: 78, 85, 62 = தரவு; "வகுப்பின் சராசரி 75" = தகவல்.

கட்டமைப்புள்ள vs கட்டமைப்பற்ற தரவு

கட்டமைப்புள்ள (Structured)கட்டமைப்பற்ற (Unstructured)
முன்வரையறுக்கப்பட்ட வடிவில், அட்டவணைகளில் (வரிசை/நிரல்)நிலையான வடிவம் இல்லாதது
எளிதில் தேடல்/பகுப்பாய்வுதேடல்/பகுப்பாய்வு கடினம்
எ.கா: மாணவர் அட்டவணை, வங்கிப் பதிவுகள்எ.கா: மின்னஞ்சல், படங்கள், காணொளி, சமூக ஊடகப் பதிவுகள்

தரவுத்தளம் ஏன்? (கோப்பு முறை vs தரவுத்தளம்)

பழைய கோப்பு அடிப்படை முறையில் (file-based) — தரவு பல கோப்புகளில் சிதறி, மீள்நிகழ்வு (redundancy), முரண்பாடு (inconsistency), பகிர்தல் சிரமம் ஏற்பட்டன. தரவுத்தளம் (Database) தரவை ஒருங்கிணைத்து, மீள்நிகழ்வைக் குறைத்து, பகிர்தலை எளிதாக்குகிறது.

தரவுத்தள மாதிரிகள் (Database Models)

மாதிரிவிளக்கம்
தட்டைக் கோப்பு (Flat File)ஒரே அட்டவணையில் அனைத்துத் தரவும்; எளிது, ஆனால் மீள்நிகழ்வு அதிகம்
படிநிலை (Hierarchical)மரக் கட்டமைப்பு; ஒரு பெற்றோர் → பல குழந்தைகள் (1:M); எ.கா: IBM IMS
வலை (Network)ஒரு குழந்தைக்குப் பல பெற்றோர் இருக்கலாம் (M:N); படிநிலையை விட நெகிழ்வானது
தொடர்புசார் (Relational)தரவு அட்டவணைகளாக (tables), சாவிகளால் இணைக்கப்படும்; மிகப் பரவலானது; எ.கா: MySQL, Oracle
பொருள்-தொடர்புசார் (Object-Relational)தொடர்புசார் + பொருள்சார் பண்புகள்; சிக்கலான தரவு வகைகளைக் கையாளும்
படிநிலை (மரம்) vs தொடர்புசார் (சாவிகளால் இணைந்த அட்டவணைகள்)
8.2தொடர்புசார் மாதிரியின் கூறுகளும் கட்டுப்பாடுகளும்

தொடர்புசார் மாதிரியில் தரவு அட்டவணைகளாக (relations/tables) அமைக்கப்படுகிறது. கீழே student அட்டவணை உதாரணம்:

student அட்டவணை (relation)
index_nonamemarkscity
1001Amuthan78Jaffna
1002Keerthana85Vavuniya
1003Nilavan62Jaffna
  • தொடர்பு/அட்டவணை (Relation/Table): முழு அட்டவணை.
  • பண்புக்கூறு (Attribute): நிரல் (column) — எ.கா: name, marks.
  • பதிவு/வரிசை (Tuple/Record): ஒரு வரிசை (row) — ஒரு மாணவரின் முழுத் தகவல்.
  • படி (Degree): பண்புக்கூறுகளின் எண்ணிக்கை (இங்கு 4).
  • எண்ணுமை (Cardinality): பதிவுகளின் எண்ணிக்கை (இங்கு 3).
  • தொடர்பு (Relationship): அட்டவணைகளுக்கிடையேயான இணைப்பு (சாவிகளால்).

கட்டுப்பாடுகள் (Constraints)

தரவின் செம்மையை (integrity) உறுதிசெய்ய விதிக்கப்படும் விதிகள்:

கட்டுப்பாடுசெயல்
NOT NULLபுலம் வெறுமையாக (NULL) இருக்கக் கூடாது — மதிப்பு கட்டாயம்
UNIQUEபுலத்தின் அனைத்து மதிப்புகளும் தனித்துவமாக இருக்க வேண்டும் (மீள்நிகழ்வு கூடாது)
PRIMARY KEYஒவ்வொரு பதிவையும் தனித்துவமாக அடையாளம் காணும் புலம் = NOT NULL + UNIQUE
FOREIGN KEYஇன்னொரு அட்டவணையின் primary key ஐச் சுட்டிக்காட்டும் புலம் — அட்டவணைகளை இணைக்கும்
CHECKபுலத்தின் மதிப்பு ஒரு நிபந்தனையைப் பூர்த்திசெய்ய வேண்டும். எ.கா: CHECK (marks >= 0 AND marks <= 100)
PRIMARY KEY vs FOREIGN KEY Primary key = ஒரு அட்டவணையில் ஒவ்வொரு வரிசையையும் தனித்துவமாக அடையாளம் காணும் (எ.கா: index_no). Foreign key = வேறு அட்டவணையின் primary key ஐச் சுட்டி, இரு அட்டவணைகளை இணைக்கும் (referential integrity).
8.3DBMS & SQL — DDL, DML

DBMS (Database Management System) = தரவுத்தளத்தை உருவாக்கி, நிர்வகித்து, அணுக உதவும் மென்பொருள் (MySQL, Oracle, SQL Server). SQL (Structured Query Language) = தரவுத்தளத்துடன் தொடர்புகொள்ளும் மொழி.

DDL (Data Definition Language) — கட்டமைப்பு கட்டளைகள்

-- அட்டவணை உருவாக்கல்
CREATE TABLE student (
    index_no INT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    marks INT CHECK (marks >= 0 AND marks <= 100),
    city VARCHAR(30)
);

-- அட்டவணையை மாற்றல் (புதிய நிரல் சேர்த்தல்)
ALTER TABLE student ADD email VARCHAR(60);

-- நிரலை நீக்கல்
ALTER TABLE student DROP COLUMN email;

-- அட்டவணையை முழுமையாக நீக்கல்
DROP TABLE student;

-- தரவுத்தளத்தை நீக்கல்
DROP DATABASE school;
DROP எச்சரிக்கை DROP TABLE அட்டவணையையும் அதன் எல்லாத் தரவையும் நிரந்தரமாக நீக்கும். DELETE தரவை மட்டும் நீக்கும் (அட்டவணை அமைப்பு எஞ்சும்); DROP அமைப்பையும் நீக்கும்.

DML (Data Manipulation Language) — தரவு கட்டளைகள்

-- 1) தரவு சேர்த்தல்
INSERT INTO student VALUES (1001, 'Amuthan', 78, 'Jaffna');

-- 2) தரவு மீட்டெடுத்தல் — அடிப்படை SELECT
SELECT * FROM student;                          -- எல்லா நிரல், எல்லா வரிசை
SELECT name, marks FROM student;                -- குறிப்பிட்ட நிரல்கள்
SELECT * FROM student WHERE marks > 75;          -- நிபந்தனை
SELECT * FROM student ORDER BY marks DESC;       -- வரிசைப்படுத்தல்
SELECT city, COUNT(*) FROM student GROUP BY city; -- குழுவாக்கம்

-- 3) தரவு திருத்தல்
UPDATE student SET marks = 80 WHERE index_no = 1003;

-- 4) தரவு நீக்கல்
DELETE FROM student WHERE index_no = 1003;

பல அட்டவணை SELECT — INNER JOIN

இரு அட்டவணைகளை அவற்றின் பொதுப் புலத்தால் (foreign key) இணைத்து தரவைப் பெறுதல். உதாரணம்: student அட்டவணையும் course அட்டவணையும்:

-- ஒவ்வொரு மாணவரின் பெயரும் அவர் கற்கும் பாடப் பெயரும்
SELECT student.name, course.course_name
FROM student
INNER JOIN course
ON student.course_id = course.course_id;
INNER JOIN INNER JOIN இரு அட்டவணைகளிலும் பொருந்தும் வரிசைகளை மட்டும் தரும். ON நிபந்தனை எந்தப் புலங்கள் பொருந்த வேண்டும் எனக் காட்டுகிறது (வழக்கமாக foreign key = primary key).
8.4ER வரைபடம் (Entity-Relationship Diagram)

ER வரைபடம் = தரவுத்தளத்தின் தர்க்க அமைப்பை வரைபடமாகக் காட்டும் கருவி. முக்கிய குறியீடுகள்:

ER குறியீடுகள்: நிறுவி (செவ்வகம்) · பண்புக்கூறு (நீள்வட்டம்) · தொடர்பு (சாய்சதுரம்)
  • நிறுவி (Entity): தரவு சேமிக்கப்படும் பொருள்/கருத்து (Student, Course). செவ்வகம்.
  • பண்புக்கூறு (Attribute): நிறுவியின் பண்பு (name, marks). நீள்வட்டம்.
  • நிறுவி அடையாளங்காட்டி (Entity Identifier / Key attribute): நிறுவியைத் தனித்துவமாக அடையாளம் காணும் பண்புக்கூறு (index_no). கோடிட்ட நீள்வட்டம்.
  • தொடர்பு (Relationship): நிறுவிகளுக்கிடையேயான இணைப்பு (Student "enrols in" Course). சாய்சதுரம்.

எண்ணளவை (Cardinality)

இரு நிறுவிகளுக்கிடையேயான தொடர்பின் வகை:

வகைவிளக்கம்எடுத்துக்காட்டு
1:1 (ஒன்றுக்கு ஒன்று)ஒரு நிறுவி ↔ ஒரே ஒரு நிறுவிஒரு நபர் ↔ ஒரு கடவுச்சீட்டு
1:M (ஒன்றுக்குப் பல)ஒரு நிறுவி ↔ பல நிறுவிகள்ஒரு வகுப்பு ↔ பல மாணவர்கள்
M:N (பலருக்குப் பல)பல நிறுவிகள் ↔ பல நிறுவிகள்பல மாணவர்கள் ↔ பல பாடங்கள்
M:N தொடர்பு உதாரணம் — மாணவர் & பாடம்

EER வரைபடம் (Enhanced ER) — அறிமுகம்

EER = ER இன் விரிவாக்கம்; கூடுதல் கருத்துகளைச் சேர்க்கிறது: பொதுமைப்படுத்தல் (generalization), சிறப்பாக்கம் (specialization), பாரம்பரியம் (inheritance), துணைவகுப்புகள் (subclasses). சிக்கலான தரவு உறவுகளை மாதிரியாக்க உதவுகிறது.

8.5தர்க்கத் திட்டம் (Logical Schema) & சாவிகள்

தொடர்புசார் திட்டம் (Relational Schema): ஒரு அட்டவணையின் கட்டமைப்பை எழுத்துருவில் காட்டுதல் — primary key கோடிடப்படும்:

student (index_no, name, marks, city)
         ↑ primary key (கோடிடப்பட்டது)

enrolment (index_no, course_id, grade)
           ↑ கூட்டு primary key
  • தொடர்பு நிகழ்வு (Relation Instance): ஒரு குறிப்பிட்ட நேரத்தில் அட்டவணையில் உள்ள உண்மையான தரவு (வரிசைகள்).
  • களம் (Domain): ஒரு பண்புக்கூறு எடுக்கக்கூடிய அனுமதிக்கப்பட்ட மதிப்புகளின் தொகுதி. எ.கா: marks இன் domain = 0–100 முழுஎண்கள்; gender இன் domain = {M, F}.

சாவிகளின் வகைகள் (Types of Keys)

சாவிவிளக்கம்
மிகை/வேட்பாளர் சாவி
(Candidate Key)		ஒரு வரிசையைத் தனித்துவமாக அடையாளம் காணக்கூடிய ஒவ்வொரு புலம்/புலத் தொகுப்பு. (பல candidate keys இருக்கலாம்.) எ.கா: index_no, NIC.
முதன்மைச் சாவி
(Primary Key)		candidate keys இல் தெரிவுசெய்யப்பட்ட ஒன்று — அட்டவணையின் அதிகாரபூர்வ அடையாளங்காட்டி. NULL இருக்கக் கூடாது.
மாற்றுச் சாவி
(Alternate Key)		primary key ஆகத் தெரிவுசெய்யப்படாத எஞ்சிய candidate keys. எ.கா: index_no primary எனில், NIC = alternate.
அயற் சாவி
(Foreign Key)		இன்னொரு அட்டவணையின் primary key ஐச் சுட்டிக்காட்டும் புலம் — அட்டவணைகளை இணைக்கும்.
கூட்டுச் சாவி
(Composite Key)		இரண்டு அல்லது அதற்கு மேற்பட்ட புலங்கள் சேர்ந்து உருவாகும் primary key.
சாவிகளின் தொடர்பு Candidate keys = தனித்துவமாக அடையாளம் காணக்கூடிய எல்லாமே. அவற்றுள் ஒன்று → Primary key; எஞ்சியவை → Alternate keys. (Primary + Alternate = Candidate.)
8.6ER → தர்க்கத் திட்ட மாற்றம் (ER to Logical Schema)

ER வரைபடத்தை தொடர்புசார் அட்டவணைகளாக மாற்றும் விதிகள்:

1) நிறுவி மாற்றம் (Entity Transformation)

ஒவ்வொரு நிறுவியும் ஒரு அட்டவணையாக மாறும்; நிறுவியின் அடையாளங்காட்டி → primary key.

2) பண்புக்கூறு மாற்றம்

ஒவ்வொரு பண்புக்கூறும் அந்த அட்டவணையின் நிரலாக மாறும். கூட்டுப் பண்புக்கூறு (composite) → தனித் தனி நிரல்களாக; பல்மதிப்புப் பண்புக்கூறு (multivalued) → தனி அட்டவணையாக.

3) தொடர்பு மாற்றம் (Relationship Transformation)

தொடர்பு வகைமாற்ற விதி
1:1ஒரு அட்டவணையின் primary key ஐ மற்றதில் foreign key ஆகச் சேர்க்க
1:M"பல" பக்க அட்டவணையில், "ஒன்று" பக்கத்தின் primary key ஐ foreign key ஆகச் சேர்க்க
M:Nஒரு புதிய இணைப்பு அட்டவணை (junction/bridge table) உருவாக்க — இரு நிறுவிகளின் primary keys ஐயும் foreign keys ஆகக் கொண்டு (சேர்ந்து composite key)
M:N — மிக முக்கியம் M:N தொடர்பை நேரடியாக அட்டவணைகளில் காட்ட முடியாது. எனவே புதிய இணைப்பு அட்டவணை உருவாக்கப்படுகிறது. எ.கா: Student(M) — Course(N) → enrolment(index_no, course_id, grade) — இதில் index_no, course_id இரண்டும் சேர்ந்து composite primary key, இரண்டும் foreign keys.
8.7இயல்நிலையாக்கம் (Normalization) — 0NF → 3NF

இயல்நிலையாக்கம் = அட்டவணைகளை மீள்நிகழ்வையும் (redundancy) முரண்பாடுகளையும் (anomalies) குறைக்கும் வகையில் முறையாகப் பிரிக்கும் செயல்முறை.

மீள்நிகழ்வு & முரண்பாடுகள் (Anomalies)

இயல்நிலையாக்கப்படாத அட்டவணையில் ஒரே தரவு மீண்டும் மீண்டும் வந்தால், மூன்று வகை முரண்பாடுகள் ஏற்படும்:

  • சேர்க்கை முரண்பாடு (Insert anomaly): தொடர்பற்ற தரவு இல்லாமல் புதிய பதிவைச் சேர்க்க முடியாமை. எ.கா: இன்னும் மாணவர் இல்லாத புதிய பாடத்தைச் சேர்க்க முடியாது.
  • புதுப்பிப்பு முரண்பாடு (Update anomaly): மீண்டும் வரும் தரவை ஒரே இடத்தில் மாற்றி, மற்ற இடங்களில் மறந்தால் முரண்பாடு. எ.கா: ஆசிரியரின் தொலைபேசி எண் பல வரிசைகளில் இருந்தால், ஒன்றை மட்டும் மாற்றினால் முரண்பாடு.
  • நீக்கல் முரண்பாடு (Delete anomaly): ஒரு பதிவை நீக்கும்போது தேவையான வேறு தரவும் இழக்கப்படுதல்.

செயற்பாட்டுச் சார்பு (Functional Dependency)

A → B எனில் "A ஐ அறிந்தால் B ஐத் தீர்மானிக்கலாம்". எ.கா: index_no → name (மாணவர் இலக்கத்தை அறிந்தால் பெயரைத் தீர்மானிக்கலாம்).

  • முழுச் சார்பு (Full): B முழு (கூட்டு) சாவியையும் சார்ந்தது.
  • பகுதிச் சார்பு (Partial): B கூட்டுச் சாவியின் ஒரு பகுதியை மட்டும் சார்ந்தது. (2NF இல் நீக்கப்படும்.)
  • கடத்துச் சார்பு (Transitive): A → B, B → C எனில் A → C (மறைமுகம்). (3NF இல் நீக்கப்படும்.)

இயல்நிலை வடிவங்கள் — படிப்படியாக

தொடக்கம் (0NF / Unnormalized): மீள்படும் குழுக்கள் (repeating groups) உள்ள அட்டவணை:

0NF — ஒரே கலத்தில் பல மதிப்புகள்
index_nonamesubjects
1001AmuthanICT, Physics, Maths
1002KeerthanaICT, Bio

1NF (முதல் இயல்நிலை): ஒவ்வொரு கலமும் ஒற்றை அணுமதிப்பு (atomic) கொண்டிருக்க வேண்டும் — மீள்படும் குழுக்கள் கூடாது:

1NF — ஒவ்வொரு கலமும் ஒற்றை மதிப்பு
index_nonamesubject
1001AmuthanICT
1001AmuthanPhysics
1001AmuthanMaths
1002KeerthanaICT
1002KeerthanaBio

(primary key = index_no + subject கூட்டுச் சாவி.)

2NF (இரண்டாம் இயல்நிலை): 1NF ஆக இருந்து, பகுதிச் சார்பு இல்லாமல் இருக்க வேண்டும் — ஒவ்வொரு சாவியற்ற பண்புக்கூறும் முழுக் கூட்டுச் சாவியையும் சார்ந்திருக்க வேண்டும். இங்கு name index_no ஐ மட்டும் சார்ந்தது (subject அல்ல) — பகுதிச் சார்பு! எனவே பிரிக்க:

student அட்டவணை
index_noname
1001Amuthan
1002Keerthana
enrolment அட்டவணை
index_nosubject

1001Physics
1002Bio

3NF (மூன்றாம் இயல்நிலை): 2NF ஆக இருந்து, கடத்துச் சார்பு இல்லாமல் இருக்க வேண்டும் — சாவியற்ற பண்புக்கூறு இன்னொரு சாவியற்ற பண்புக்கூற்றைச் சார்ந்திருக்கக் கூடாது. உதாரணம்: student(index_no, name, city, city_postcode) — இங்கு index_no → city → city_postcode (கடத்து!). எனவே பிரிக்க:

student

மூன்று வடிவங்களும் — சுருக்கம் 1NF: ஒற்றை அணுமதிப்பு (மீள்படும் குழு கூடாது). 2NF: 1NF + பகுதிச் சார்பு இல்லை (முழுச் சாவியைச் சார்தல்). 3NF: 2NF + கடத்துச் சார்பு இல்லை.
நினைவுச் சொற்றொடர்: "The key, the whole key, and nothing but the key" — 1NF (key உள்ளது), 2NF (whole key), 3NF (nothing but the key).
16சூழ்நிலை அடிப்படையிலான வினாக்கள் (விடைகளுடன்)

பின்வரும் வினாக்கள் ஒரு நூலக (library) மற்றும் மருத்துவமனை (hospital) சூழல்களை அடிப்படையாகக் கொண்டவை. முதலில் முயன்று, பின் விடையைச் சரிபார்க்கவும்.

வினா 01
ஒரு பாடசாலை, மாணவர்களின் பெயர்களையும் முகவரிகளையும் ஒரு Excel தாளில் வைத்துள்ளது. ஒவ்வொரு வகுப்பாசிரியரும் தனிப் பிரதி வைத்துள்ளனர்.

இந்த கோப்பு அடிப்படை முறையின் இரண்டு குறைபாடுகளைக் கூறி, தரவுத்தளம் எவ்வாறு தீர்க்கும் எனக் கூறுக.

விடை

குறைபாடுகள்: தரவு மீள்நிகழ்வு (பல பிரதிகள்); முரண்பாடு (ஒரு பிரதியில் மாற்றி மற்றதில் மறந்தால்); பகிர்தல்/புதுப்பித்தல் சிரமம்.

தரவுத்தளத் தீர்வு: தரவை ஒரே மைய இடத்தில் சேமித்து, மீள்நிகழ்வைக் குறைத்து, அனைவரும் ஒரே துல்லியமான தரவைப் பகிர்ந்து அணுக முடியும்.

வினா 02
ஒரு நூலகத்தில் book அட்டவணையில் book_id, title, author, price ஆகிய புலங்கள் உள்ளன.

(அ) இந்த அட்டவணையில் எத்தனை பண்புக்கூறுகள் (degree)? (ஆ) book_id க்கு ஏற்ற கட்டுப்பாடு எது, ஏன்?

விடை

(அ) படி (degree) = 4 (book_id, title, author, price).

(ஆ) book_id க்கு PRIMARY KEY — ஏனெனில் ஒவ்வொரு நூலையும் தனித்துவமாக அடையாளம் காண வேண்டும் (NOT NULL + UNIQUE).

வினா 03
நூலகத்தில் book(book_id, title, price) அட்டவணையை உருவாக்க வேண்டும். price 0 க்கு குறைவாக இருக்கக் கூடாது.

இந்த அட்டவணையை உருவாக்கும் SQL DDL கட்டளையை எழுதுக.

விடை 
CREATE TABLE book (
    book_id INT PRIMARY KEY,
    title VARCHAR(100) NOT NULL,
    price DECIMAL(8,2) CHECK (price >= 0)
);
வினா 04
book அட்டவணையில் விலை ரூ.1000 க்கு மேற்பட்ட நூல்களின் தலைப்புகளை விலையின் இறங்குவரிசையில் பெற வேண்டும்.

இதற்கான SQL SELECT கட்டளையை எழுதுக.

விடை 
SELECT title, price
FROM book
WHERE price > 1000
ORDER BY price DESC;
வினா 05
நூலகத்தில் member(member_id, name) மற்றும் loan(loan_id, member_id, book_id) அட்டவணைகள் உள்ளன.

ஒவ்வொரு கடன் பதிவுக்கும் உறுப்பினரின் பெயரைக் காட்டும் SQL கட்டளையை (INNER JOIN பயன்படுத்தி) எழுதுக.

விடை 
SELECT loan.loan_id, member.name, loan.book_id
FROM loan
INNER JOIN member
ON loan.member_id = member.member_id;
loan.member_id (foreign key) = member.member_id (primary key) — இதுவே இணைப்பு நிபந்தனை.
வினா 06
member_id 205 உறுப்பினரின் பெயரை "Ravi" ஆக மாற்ற வேண்டும்.

இதற்கான SQL கட்டளையை எழுதுக. WHERE விட்டால் என்ன நடக்கும்?

விடை 
UPDATE member SET name = 'Ravi'
WHERE member_id = 205;

WHERE விட்டால் — எல்லா உறுப்பினர்களின் பெயரும் "Ravi" ஆக மாறிவிடும்!

வினா 07
ஒரு மருத்துவமனையில் ஒரு நோயாளர் (Patient) பல சந்திப்புகளை (Appointment) மேற்கொள்கிறார்; ஒரு சந்திப்பு ஒரே நோயாளருக்கானது.

Patient க்கும் Appointment க்கும் இடையேயான தொடர்பின் எண்ணளவை (cardinality) என்ன?

விடை

1:M (ஒன்றுக்குப் பல) — ஒரு நோயாளர் பல சந்திப்புகளை மேற்கொள்ளலாம், ஆனால் ஒரு சந்திப்பு ஒரே நோயாளருக்கானது.

வினா 08
ஒரு மருத்துவமனையில் ஒரு மருத்துவர் பல நோயாளர்களைப் பார்க்கிறார்; ஒரு நோயாளர் பல மருத்துவர்களைப் பார்க்கலாம்.

இந்த M:N தொடர்பை அட்டவணைகளாக மாற்றும்போது என்ன செய்ய வேண்டும்? அட்டவணை அமைப்பைக் காட்டுக.

விடை

M:N தொடர்பை நேரடியாகக் காட்ட முடியாது — ஒரு புதிய இணைப்பு அட்டவணை உருவாக்க வேண்டும்:

doctor(doctor_id, name)
patient(patient_id, name)
consultation(doctor_id, patient_id, date)
   ↑ doctor_id + patient_id = composite primary key
     இரண்டும் foreign keys
வினா 09
ஒரு அட்டவணையில் index_no மற்றும் NIC ஆகிய இரண்டுமே ஒவ்வொரு மாணவரையும் தனித்துவமாக அடையாளம் காணக்கூடியவை. index_no primary key ஆகத் தெரிவுசெய்யப்பட்டது.

(அ) candidate keys யாவை? (ஆ) NIC இன் சாவி வகை என்ன?

விடை

(அ) Candidate keys = index_no மற்றும் NIC (இரண்டும் தனித்துவமாக அடையாளம் காணக்கூடியவை).

(ஆ) index_no primary key ஆனதால், NIC = alternate key (மாற்றுச் சாவி).

வினா 10
பின்வரும் அட்டவணை: order(order_id, customer_name, customer_phone, item). ஒரே வாடிக்கையாளர் பல ஆர்டர்களைச் செய்வதால், அவரது தொலைபேசி எண் பல வரிசைகளில் மீண்டும் வருகிறது.

இங்கு ஏற்படக்கூடிய புதுப்பிப்பு முரண்பாட்டை (update anomaly) விளக்குக.

விடை

வாடிக்கையாளரின் தொலைபேசி எண் மாறினால், அது பல வரிசைகளில் உள்ளது. சில வரிசைகளில் மட்டும் மாற்றி, மற்றவற்றில் மறந்தால் — ஒரே வாடிக்கையாளருக்கு வெவ்வேறு எண்கள் தோன்றி தரவு முரண்பாடு ஏற்படும். (தீர்வு: வாடிக்கையாளர் தரவைத் தனி அட்டவணையாகப் பிரித்தல் — normalization.)

வினா 11
அட்டவணை: student(index_no, name, subjects) இல் subjects கலத்தில் "ICT, Physics, Maths" எனப் பல மதிப்புகள் உள்ளன.

இது எந்த இயல்நிலையில் (NF) உள்ளது? 1NF க்கு மாற்ற என்ன செய்ய வேண்டும்?

விடை

இது 0NF (unnormalized) — ஒரே கலத்தில் பல மதிப்புகள் (மீள்படும் குழு).

1NF க்கு: ஒவ்வொரு பாடத்துக்கும் தனி வரிசை உருவாக்கி, ஒவ்வொரு கலமும் ஒற்றை அணுமதிப்பு (atomic) கொண்டிருக்க வேண்டும். primary key = index_no + subject.

வினா 12
அட்டவணை: enrolment(index_no, course_id, student_name, course_fee) — primary key = (index_no + course_id). student_name index_no ஐ மட்டும் சார்ந்தது; course_fee course_id ஐ மட்டும் சார்ந்தது.

இங்குள்ள பகுதிச் சார்பை (partial dependency) விளக்கி, 2NF க்கு மாற்றுக.

விடை

பகுதிச் சார்பு: student_name முழுக் கூட்டுச் சாவியை (index_no+course_id) அல்ல, index_no ஐ மட்டும் சார்ந்தது; course_fee course_id ஐ மட்டும் சார்ந்தது. இவை பகுதிச் சார்புகள்.

2NF — பிரிக்க:

student(index_no, student_name)
course(course_id, course_fee)
enrolment(index_no, course_id)
வினா 13
அட்டவணை: employee(emp_id, name, dept_id, dept_name). emp_id → dept_id, dept_id → dept_name.

இங்குள்ள கடத்துச் சார்பை (transitive dependency) விளக்கி, 3NF க்கு மாற்றுக.

விடை

கடத்துச் சார்பு: emp_id → dept_id → dept_name. எனவே dept_name emp_id ஐ மறைமுகமாக (transitively) சார்ந்துள்ளது.

3NF — பிரிக்க:

employee(emp_id, name, dept_id)
department(dept_id, dept_name)
வினா 14
ஒரு நூலகத்தில் இன்னும் யாரும் கடன் வாங்காத ஒரு புதிய நூலைச் சேர்க்க வேண்டும். ஆனால் அட்டவணை அமைப்பு loan & book தகவலை ஒரே அட்டவணையில் வைத்துள்ளது.

இங்கு ஏற்படும் முரண்பாட்டின் பெயர் என்ன? விளக்குக.

விடை

சேர்க்கை முரண்பாடு (Insert anomaly) — கடன் தகவல் (member_id, loan_date) இல்லாமல் புதிய நூலைச் சேர்க்க முடியாது, ஏனெனில் அவை ஒரே அட்டவணையில் உள்ளன. (தீர்வு: book மற்றும் loan ஐத் தனி அட்டவணைகளாகப் பிரித்தல்.)

வினா 15
book அட்டவணையில் "out of print" ஆன நூல்களை நீக்க வேண்டும். book_id 110 ஐ நீக்கவும்.

SQL கட்டளையை எழுதுக. DELETE க்கும் DROP க்கும் உள்ள வேறுபாடு என்ன?

விடை 
DELETE FROM book WHERE book_id = 110;

DELETE: அட்டவணையிலிருந்து குறிப்பிட்ட வரிசைகளை மட்டும் நீக்கும்; அட்டவணை அமைப்பு எஞ்சும்.

DROP TABLE: அட்டவணையையும் அதன் முழுத் தரவையும் அமைப்பையும் நிரந்தரமாக நீக்கும்.

வினா 16
ஒரு கடையில் ஒவ்வொரு நகரத்துக்கும் (city) ஒரே அஞ்சல் குறியீடு (postcode) உள்ளது. அட்டவணை: customer(cust_id, name, city, postcode).

(அ) இங்குள்ள சார்பு வகை என்ன? (ஆ) 3NF க்கு மாற்றிய அட்டவணைகளைக் காட்டுக.

விடை

(அ) cust_id → city → postcode = கடத்துச் சார்பு (transitive).

(ஆ) 3NF:

customer(cust_id, name, city)
city(city, postcode)
